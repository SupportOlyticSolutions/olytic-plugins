# Aule Core Standards

**Version:** 1.0.0
**Owner:** Aulë
**Applies to:** All Olytic plugins — every plugin installed in the workspace

This document defines the organization-wide standards that all Olytic plugins must conform to. It is enforced by `aule-verifier` during compliance scans and applied by `trashbot` during ecosystem sweeps. It is not a suggestion — plugins that fail these standards are flagged as non-compliant.

---

## 1. Plugin Identity

Every plugin must have a `.claude-plugin/plugin.json` file with all required fields populated. The authoritative field list is available at runtime via `invoke skill: olytic-core-schemas / schema: plugin-identity`.

**Required fields:**
- `name` — kebab-case, unique across the workspace
- `version` — semver (major.minor.patch)
- `description` — one sentence, ≤120 characters
- `author.name` and `author.email`
- `keywords` — 1–10 terms

**Validation rules:**
- `name` must match the plugin's folder name and `src-[name]` source folder
- `version` must be incremented on every release — downgrade is rejected
- `connectors` declared in `plugin.json` must be a subset of what is declared in `workspace.json`

---

## 2. Folder Structure

Every plugin must follow the canonical two-level structure:

```
Plugins/[plugin-name]/
├── src-[plugin-name]/          # All plugin source files
│   ├── .claude-plugin/
│   │   └── plugin.json
│   ├── skills/
│   │   ├── [plugin-name]-telemetry/
│   │   │   └── SKILL.md
│   │   └── [other skills]/
│   │       └── SKILL.md
│   ├── agents/                 # Optional
│   ├── commands/               # Optional
│   ├── hooks/                  # Required if hooks declared in plugin.json
│   │   └── hooks.json
│   └── README.md
└── [plugin-name].plugin        # Packaged zip — built by compiler
```

**Rules:**
- Source folder is always `src-[plugin-name]` — no exceptions
- `.plugin` file lives in the parent folder, never inside `src-[plugin-name]/`
- No flat files in the plugin parent folder other than the `.plugin` archive

---

## 3. Required Components

Every plugin must include:

- **Telemetry skill** — `[plugin-name]-telemetry` — emits structured events to olytic-gateway
- **README.md** — inside `src-[plugin-name]/` — must include a Claude OS Identity section describing the plugin's purpose, components, constraints, and memory scope
- **Session summarizer skill** — `[plugin-name]-session-summarizer` — fires on SessionClose, writes structured session record to vault via olytic-gateway

The telemetry skill and session summarizer are auto-generated by the compiler. They must not be manually edited to remove their gateway integration.

---

## 4. Hook Declarations

If a plugin uses hooks:

- `hooks.json` must be at `src-[plugin-name]/hooks/hooks.json`
- The `hooks` field in `plugin.json` must point to `hooks/hooks.json`
- Hook types must be valid platform hook names: `PostToolUse`, `UserPromptSubmit`, `PreToolUse`, `PostModelResponse`, `SessionClose`
- Each hook entry must have a `type` field (`prompt` or `filter`) and a `prompt` or `filter` value

---

## 5. Connector Declarations

- Every connector used by the plugin must be declared in `plugin.json` under `connectors`
- `olytic-gateway` is auto-injected by the compiler — do not declare it manually in specs
- Connector `id` values must match entries in `workspace.json` connectors
- `scopes` must be the minimum required — no wildcard scopes

---

## 6. Telemetry Standard

All telemetry events must conform to the `telemetry-event` schema. Fetch at runtime via `invoke skill: olytic-core-schemas / schema: telemetry-event`. The 8 canonical event types are:

`skill_invoke`, `decision_trace`, `feedback`, `violation`, `not_found_reported`, `verification_gate`, `permission_gate`, `agent_trigger`

Events are emitted via the `log_telemetry` tool on the `olytic-gateway` MCP connector. Each event must include `timestamp`, `event`, `plugin`, `plugin_version`, `platform`, and `component` fields at minimum. Never include `org_id`, `user_id`, or `client_id` — gateway injects them server-side.

---

## 7. Memory and Vault

Vault writes must conform to the `vault-entry` schema. Fetch at runtime via `invoke skill: olytic-core-schemas / schema: vault-entry`. Rules:

- `entry_type` must be one of: `session_summary`, `user_preference`, `decision_record`, `correction`, `business_fact`, `plugin_feedback` — underscores, matching `memory_entries.entity_type` exactly
- `loop` must be one of: `plugin`, `user`, `client` — routes the entry into the correct refinement loop
- `platform` is a build-time constant — hardcoded at generation time, never inferred at runtime
- `org_id`, `client_id`, and `user_id` are never included in the payload — injected server-side by the gateway
- Only the session summarizer skill may write `session_summary` entries
- No plugin may read vault entries belonging to a different plugin without explicit cross-plugin access declared in `workspace.json`

---

## 8. Versioning

- Plugins follow semver: `major.minor.patch`
- Patch bump: bug fixes, copy changes, non-breaking skill edits
- Minor bump: new components, new connectors, new hooks
- Major bump: breaking changes to trigger phrases, component removals, connector scope reductions

The compiler stamps the version from the spec. Do not manually edit version in `plugin.json` after compilation — update the spec and recompile.

## Enforcement

| Check | Tool |
|---|---|
| Identity fields present and valid | `aule-verifier` |
| Folder structure correct | `aule-verifier` |
| Telemetry + session-summarizer present | `aule-verifier` |
| Hook declarations valid | `aule-verifier` |
| Connector declarations match workspace | `aule-verifier` |
| Schema compliance | `aule-verifier` |
| Auto-remediation of failing plugins | `trashbot` |

`aule-verifier` runs in detect-only mode — it never writes. `trashbot` applies fixes after verifier produces a compliance report.
