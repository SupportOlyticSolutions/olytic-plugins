How to Design Claude Cowork Plugins to Work as a Team

The Hats Framework for designing Claude Cowork plugins as organizational roles that collaborate automatically without explicit commands.

Published March 2026 · Olytic Solutions

The core insight: The most effective Claude Cowork plugins are designed as organizational roles, not feature bundles. The Hats Framework treats each plugin as a mastery-level team member with domain knowledge, skills, judgment, relationships to other hats, and defined escalation paths. This architecture transforms isolated tools into a collaborative system where relevant perspectives show up automatically, without explicit commands.

The Hats Framework

Most people build Claude Cowork plugins the way they'd build a tool. A content plugin. A data plugin. A calendar plugin. Feature bundles organized by function.

This is wrong. Or at least, it's incomplete in a way that limits what you can achieve.

We learned this the hard way. We built Magneto and The One Ring as feature plugins first. Everything worked until we tried to use them together. Magneto would create content, you'd manually invoke The One Ring to check brand compliance, and they'd act like they'd never met. We spent two weeks debugging what should have been tight collaboration. The fix: one line in a skill description telling Magneto to load The One Ring before proceeding. One line. Two weeks. That's the architecture lesson right there.

Here's a better mental model: every plugin is a hat.

Not "wearing many hats," the frantic multitasking version, a hat like an organizational role. The marketing hat. The compliance hat. The product strategy hat. Each one contains all the knowledge, skills, workflows, judgment, and institutional memory that role requires to operate at a mastery level.

Design plugins this way and you stop thinking about what the plugin does and start thinking about who it is. That shift matters more than it sounds.

How Should You Design Claude Cowork Plugins to Work as a Team?

The right mental model for Claude Cowork plugins isn't tools. It's organizational roles. Here's the framework we use to design them. Consider the difference:

Tool-Oriented Plugin Hat-Oriented Plugin
Identity "Content creation tool" "The content strategist"
Scope Features and functions Judgment, knowledge, and workflows
Improvement Add more features Refine toward the platonic ideal of the role
Interaction User invokes specific commands Claude recognizes when this perspective is needed
Design question "What should this do?" "What would this person know, and when would you ask them?"

Tool model: user picks a tool. Hat model: hats recognize when they're needed.

The hat model gives you something the tool model can't: a plugin that knows when to speak up. A content creation tool waits to be invoked. A content strategist recognizes when a conversation has content implications and weighs in, the same way a colleague would raise a hand in a meeting.

What a Well-Designed Hat Actually Contains

If you're going to build plugins as organizational roles, each hat needs more than a list of skills. It needs the full picture of what someone in that role carries around in their head.

We call this the Five Layers of a Mastery-Level Plugin.

1 Domain Knowledge

The foundational expertise. For a brand standards hat, this is the full brand guide: voice rules, messaging pillars, competitive positioning, design preferences. Not a summary. The whole thing. A brand manager doesn't work from bullet points; they have the standards internalized. Your plugin should too.

2 Skills and Workflows

The things this role actually does. A content strategist doesn't just "know about content." They produce briefs, audit existing pages, research competitors, evaluate GEO performance. Each of these is a discrete skill with its own inputs, process, and outputs. Map them explicitly.

3 Judgment and Opinions

This is where a mediocre plugin becomes a useful one. A brand standards hat shouldn't just flag violations. It needs taste. Real opinions. "We don't write like this because it sounds like a SaaS vendor trying too hard." "That headline is technically correct but it reads like a press release." A senior person in this role would have a point of view. Encode that point of view. That's what separates judgment from just flagging rules.

4 Relationships to Other Hats

Real organizational roles don't operate in isolation. Your brand standards hat should know that when content is being created, it has a stake. Your finance hat should know that when pricing is being discussed, it needs to weigh in. These relationships are the connective tissue of your plugin ecosystem.

5 Escalation Paths

What does this hat do when it encounters something outside its domain? A good content strategist doesn't make security decisions. They flag them for the security team. Your plugins should do the same: define what's in scope, what triggers a handoff, and where to send it.

We built our own plugin ecosystem this way at Olytic. Our corporate governance and alignment hat handles the full stack of organizational standards: voice, strategy, values, HR norms, and security policies. Not a "brand checking tool." The constitutional layer that every other hat operates under. We called it The One Ring. It rules them all.

Our content strategist hat defers to it on every piece it creates. It produces briefs, audits pages, evaluates competitive positioning, and knows exactly what voice it does not have the authority to define. When the content hat creates anything, it loads The One Ring first, the same way a competent content lead checks with brand before publishing.

That deference is what makes the system work. It is also the part that breaks first. We discovered this the hard way. We had the content hat creating and The One Ring validating, but they were not talking to each other. You would create something, the content hat would draft it fine, and then you had to manually invoke governance to check brand compliance. Two plugins on the same machine that had never been introduced. We spent two weeks wondering why governance was not catching tone issues before we realized the obvious: the content skill had no idea The One Ring existed. The fix was one line in the skill description. The time we wasted was not.

The right way to think about it: one hat creates, one hat validates, one hat learns and improves. When your plugins embody these roles, you are building a closed-loop system, not a feature collection. Humans stay in control at every decision point. The hats do the heavy lifting.

Designing Your Plugin Scheme

Individual hats matter. But the scheme, which hats exist, how they relate, where they overlap, matters more.

Think of it the same way you'd think about organizational design. You wouldn't hire a team without thinking about which roles you need, where responsibilities live, and how decisions get made when two roles disagree. Plugin scheme design is organizational design for AI.

Three Questions to Start With

1. What are the functions your organization needs to execute well?

Skip "what tools would be nice" and ask: what functions? Brand governance, content strategy, sales operations, finance oversight, product management, security. Each function that needs consistent, high-quality judgment is a candidate for a hat.

2. Where do these functions overlap, and where should they stay separate?

The hard part: if your content hat and your brand hat both have opinions about voice, who wins? If your product hat and your finance hat disagree on pricing, how does that get resolved?

In a real organization, this is settled by authority structures and meeting culture. In a plugin ecosystem, it's settled by:

  • Explicit authority declarations. Our brand standards skill states plainly: "If there's a conflict between conversational context and these standards, the standards take precedence." That's a constitutional rule, not a suggestion.
  • Skill descriptions that encode awareness of other hats. Our content hat's skills know The One Ring exists. They reference it by name. When the content hat creates something, it loads brand standards first, because its skill instructions say to.
  • Shared reference documents. When two hats need to agree on something, give them the same source of truth. One document, two readers.

3. What's the "meeting culture" of your plugin ecosystem?

In a healthy organization, cross-functional decisions happen in meetings. Someone describes a situation, the relevant people weigh in, and you converge on an aligned decision.

A Cowork chat should function exactly like that meeting.

You type a message. Claude's routing looks at the message and determines which hats are relevant. Those hats contribute their perspective. You get a response that reflects multiple viewpoints, without having to invoke each plugin individually.

That's the theory, anyway.

The Cross-Functional Meeting Problem

Can Claude Plugins Invoke Each Other?

In Claude's architecture, plugins are namespaced and self-contained. A skill in one plugin cannot natively invoke a skill in another plugin. When you're working inside one plugin's context, the other plugins aren't automatically listening.

This means: if you build a governance-check skill inside your brand standards plugin, and then start creating a document using your content plugin, the governance check doesn't automatically fire.

It's not broken. It's by design. Plugins are isolated to prevent conflicts and maintain composability.

Architecturally, this makes sense. You don't want every plugin stepping on every other plugin's toes. But it creates a problem for the hats model, where the whole point is that relevant perspectives show up without being explicitly summoned.

Why Natural Language Routing Is Unreliable for Governance

Claude can trigger skills based on natural language. It reads skill descriptions and decides which ones are relevant to the current conversation. This is called model-invoked triggering, and it works well for straightforward cases. Ask a question about content strategy, and the content strategy hat shows up.

But model-invoked triggering is probabilistic, not deterministic. It depends on:

  • How well the skill description matches the conversational context
  • How many skills are competing for relevance at any given moment
  • Whether the model interprets the conversation as falling within the skill's scope

For creative or advisory tasks, probabilistic is fine. You want Claude to use judgment about when the content strategist hat is relevant.

For governance (compliance checks, brand validation, security reviews), probabilistic is not acceptable. Governance must happen every time, not most of the time. "Our brand check ran on 80% of published content" is not a compliance posture. It's a liability.

This is where the hats model fights itself. You want fluid, intelligent routing, hats showing up when they're needed. But governance can't be fluid. Governance has to be guaranteed. Solving that tension is the actual design problem.

How to Make Routing Actually Work

There are four architectural decisions that close the gap between "plugins as isolated tools" and "plugins as a collaborative team." Each one operates at a different level.

1. Skill Descriptions Are Your Routing Instructions

Most people underinvest here, and it costs them. The description field in a skill's configuration isn't a marketing blurb. It's a routing instruction to Claude. It tells the model: "Here's when you should bring me into the conversation."

The best plugin skill descriptions aren't documentation. They're routing instructions. Weak descriptions let skills go unnoticed. Effective descriptions are written as imperative prompts to Claude: list specific trigger phrases, declare broad authority, establish sequencing with language like "before" and "whenever."

Weak description

"Brand compliance checking tool."

Effective description

"Use this skill when the user asks to write content, create a page, draft a blog post, build a landing page, check brand compliance, review tone, write in our voice, draft an email, write a proposal. Load this skill before creating or reviewing ANY external-facing content."

The effective description does three things: lists specific trigger phrases, declares broad authority ("ANY external-facing content"), and uses the word "before" to establish sequencing. This is prompt engineering applied to plugin architecture.

2. Encode Cross-Hat Awareness Inside Each Hat

Since plugins can't invoke each other natively, teach each hat about the other hats.

In our system, our content hat's skill explicitly states: "Brand voice, ICP, and messaging rules come from The One Ring's brand-standards skill. Load that too." A direct instruction to Claude, baked into the skill's own prompt. No external routing required. The dependency is handled internally.

Pattern: every skill that creates or reviews content should include a line that says "Before proceeding, also load [related skill] for [specific context]." This turns cross-plugin collaboration from a routing problem into a prompt engineering problem. Prompt engineering is something you can control.

3. Use Hooks for Deterministic Governance

For anything that must happen, not "should" happen but must, use hooks instead of skill descriptions.

Hooks are event-driven and deterministic. They fire based on tool usage events (file written, edit made, command executed), not conversational context. A PostToolUse hook on Write or Edit will fire every time a file is created or modified, regardless of which plugin initiated the action.

Governance belongs here, in a hook that fires every time, unconditionally, regardless of which plugin initiated the action.

Governance that fires 80% of the time isn't compliance, it's a liability. If your governance mechanisms are probabilistic, you've left gaps in your controls. Hooks make governance deterministic: it happens every time, not "most of the time."

The hats model still works here. The governance hat just operates through hooks in addition to skills. Think of it as the difference between a team member who joins meetings when relevant (skill-based) and a team member who reviews every document before it ships (hook-based). Both are valid. Both are needed.

4. Use Managed CLAUDE.md for Constitutional Rules

Some rules shouldn't live in any single plugin. They're organizational-level. "All external content must pass brand review." "No client data in AI prompts." "Use this voice guide for all communications."

These belong in a managed CLAUDE.md file: an organization-level instruction set that loads before any conversation starts, across all plugins, and cannot be overridden by individual settings.

Managed CLAUDE.md is your organizational constitution. It's not a hat. It's the rules that all hats follow, that no single hat can override. When every plugin, every hat, and every conversation loads from the same constitutional rules, governance becomes systemic instead of fragmented.

Mechanism When It Fires Best For Reliability
Skill descriptions Claude judges relevance Advisory, creative, analytical hats Probabilistic
Cross-hat references When parent skill loads Dependencies between hats Reliable (when parent triggers)
Hooks On specific tool events Governance, compliance, validation Deterministic
Managed CLAUDE.md Every conversation, always Organizational-level rules Constitutional

Four mechanisms, one spectrum. Use all four. The right tool depends on whether you need judgment or certainty.

A well-designed plugin ecosystem uses all four. The mistake is assuming any single mechanism handles everything.

How to Interact with the Room

So you've designed your hats. You've set up your scheme. Now: how should you talk in a Cowork chat to get the cross-functional meeting experience?

Write Like You're Briefing a Room

The best results come from describing situations, not issuing commands. Compare:

Command-style

"Use the content plugin to write a blog post about AI readiness."

Briefing-style

"We need a blog post about AI readiness for our ICP. It should demonstrate expertise, align with our brand voice, and be optimized for AI search engines."

The command version locks you into one plugin. The briefing version describes the situation in a way that naturally invokes multiple hats. Content strategy, brand standards, and GEO optimization all have a stake.

Think about how you'd describe a project in a cross-functional meeting. You wouldn't say "Marketing, do this thing." You'd describe the situation, the goal, and the constraints, and let each function identify where they can contribute. Same principle.

When to Use Slash Commands vs. Natural Language

Slash commands (/plugin:skill) are explicit invocations. Use them when:

  • You want a specific skill's output: a content brief, a brand check, a competitive snapshot
  • You need to guarantee a particular hat weighs in, rather than hoping routing picks it up
  • You're running a governance or validation step after a creative step

Natural language is implicit routing. Use it when:

  • You're working through a multi-faceted problem where you want Claude to judge which perspectives are relevant
  • You're iterating on something and want the right hats to join as the conversation evolves
  • You trust your skill descriptions to handle routing correctly

The practical pattern we've landed on, which we call Briefing-then-Governance: start with natural language, finish with slash commands. Let the hats collaborate on the creative and strategic work. Then explicitly invoke the governance hats for the final check. This mirrors how a real team works. Brainstorm together, then run the deliverable through compliance before shipping.

The Long Game

The hats model has a compounding advantage the tools model does not. Every time a hat produces work, you can evaluate it. Did the brand hat catch that voice issue? Did the content hat's brief include the right competitive context? Did the governance hat flag the right risks? Each evaluation is an opportunity to refine. Sharpen the knowledge base. Tune the skill descriptions. Encode new opinions. Over time, each hat converges toward the platonic ideal of that organizational role. Quality compounds instead of decaying. You are not building tools that rot. You are building institutional knowledge that gets better with every use.

Build a Team, Not a Toolbox

If you're designing Claude plugins and want them to feel like a team, with hats that know their domain, recognize when to speak up, and defer to each other when appropriate, we've been through the design process ourselves.
