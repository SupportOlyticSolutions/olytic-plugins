01  ·  The Problem

The Problem: AI Systems Forget

Every AI session starts from zero. The model has no memory of the client's business, no record of past decisions, no awareness of standing preferences.

The user re-explains context they explained last month. The model re-learns what it was already told. Nothing accumulates. Nothing improves on its own. And the firm pays — in time, in inconsistency, and in the slow erosion of quality that no one can easily attribute to a single cause.

Why This Matters to a Professional Services Firm

For a managing partner or operations leader, AI is only valuable if it produces consistent, trustworthy output at scale. A tool that requires re-briefing every session is not a productivity multiplier — it is a productivity tax. Consider what gets lost:

  • Client context — preferences, deal history, communication style, standing instructions. Rebuilt from scratch each session, or forgotten entirely.
  • Institutional standards — how the firm writes, prices, structures deliverables, and speaks to clients. Not enforced. Drifting with every user and every session.
  • Decision continuity — commitments made last quarter, approaches agreed last month, corrections applied last week. Gone. The AI doesn't know, and it doesn't know it doesn't know.

The compounding effect is what makes this serious. Every session lost is a session that didn't add to the knowledge base. Every inconsistency is a quality risk. Every re-briefing is a senior professional spending time on overhead instead of work. At scale — across dozens of employees and hundreds of client engagements — this is a material problem.

The standard that matters: a client's AI system should know more about their business at the end of month twelve than it did at the end of month one — automatically, without anyone doing anything differently.

Plugins Help — But Don't Solve This

The natural instinct is to build plugins. Package the prompts, give the AI instructions, deploy it to users. And plugins do help — they standardize inputs, encode workflows, and reduce the variance in how AI is used day to day.

But plugins alone cannot solve the memory problem. Here's why:

  • No persistent state. A plugin carries instructions, not memory. When the session ends, everything learned in that session is discarded. The next session starts identically to the one before it — regardless of what happened.
  • No enforced rules. A plugin can suggest how to behave. It cannot enforce it. There is no mechanism to define which constraints are absolute (hard rules — never violate) versus advisory (soft rules — flag and proceed). Without that distinction, every rule is effectively a suggestion.
  • No schema for knowledge. Even when a plugin captures something worth saving, there is no standard for what to save, how to classify it, where to store it, or how to retrieve it later. Each plugin is an island.
  • No cross-plugin coherence. A firm may deploy multiple plugins across different functions — business development, client delivery, internal operations. Each starts blind. None knows what the others have learned. The firm's institutional knowledge never consolidates.

The result is a collection of well-intentioned tools that are, individually, stateless. The AI gets smarter in the moment and resets overnight.

This is not a model limitation. It is an architecture failure. The solution is not a better plugin — it is a designed system that gives every plugin the institutional memory it needs to compound over time.

02  ·  The Answer

The Answer: A Designed System That Remembers

The solution is not a smarter model. It is a governed architecture — a set of enforced design standards applied to every AI solution Olytic builds, so that memory compounds automatically across every plugin, every session, and every client engagement.

Step 1 — A Universal Design Pattern, Enforced

Olytic defines a single design standard — a proprietary design standard called Aule Axiom — that governs how every AI solution we build is structured. Not recommended. Enforced. Every plugin, every agent, every workflow we deploy is built to this standard.

The pattern mandates how plugins initialize, how they read and write to memory, what rules are absolute versus advisory, and how sessions are opened and closed. This is not a style guide — it is an architectural contract. A plugin that doesn't conform to Aule Axiom doesn't ship.

Aule Axiom is Olytic's proprietary architectural standard that sits above any individual model provider. It is compatible with every major platform — Claude, ChatGPT, Microsoft Copilot, and Gemini — because it is defined at the protocol level, not the prompt level. Aule Axiom-compliant solutions work across model providers without redesign.

Step 2 — Rules That Are Actually Enforced

Every Aule Axiom-compliant plugin declares its constraints explicitly — and those constraints are enforced at runtime, not just expressed in a system prompt. The protocol distinguishes between:

  • Hard rules — absolute. If the AI's response would violate one, the output is automatically corrected before it reaches the user. No exceptions.
  • Soft rules — advisory. Violations are flagged and surfaced to the user for a decision. The system doesn't override — it alerts.
  • Contextual rules — conditional. Applied only when defined criteria are met. The plugin knows when they apply.

This is what plugins have always been missing: a mechanism to enforce the standards they claim to follow. Aule Axiom provides it.

Step 3 — Session Memory Captured Automatically

At the end of every session, the system doesn't just discard the conversation. Before the ephemeral context is cleared, a structured compression process extracts what matters — decisions made, corrections applied, new facts surfaced, standing preferences confirmed — and writes it to the client's private memory store.

This is transmitted via the Olytic Gateway — Olytic's proprietary API layer — directly to the client's dedicated database. The Gateway handles the translation to each platform's native protocol, whether that is MCP tool-calling for Claude, Actions for ChatGPT, connectors for Copilot Studio, or extensions for Gemini. The client never has to do anything. The memory accumulates as a byproduct of normal usage.

Step 4 — Every Plugin Starts by Remembering

When a plugin is activated — before the user types anything — it runs its bootstrap sequence. The first action is always a fetch: retrieve the relevant memory from the client's vault, inject it into the context, and initialize from a state of knowing.

The plugin doesn't start fresh. It starts from everything that has been remembered about this client, their business, their preferences, and their history. Session after session, that foundation grows. The AI doesn't just get better at its task — it gets better at this client's version of that task.

The net effect: a system where each session makes the next one better. Not because the model improved. Because the memory did.

The sections that follow define the architecture underlying this system: the three-tier memory model, the anti-drift protocols, the Aule Axiom standard, and the enforcement layer that keeps everything consistent.

03  ·  Aule Axiom

Aule Axiom

Each AI platform defines its own transport layer — how an agent communicates with external tools. MCP handles this for Claude, Actions handle it for ChatGPT, connectors for Copilot Studio, and extensions for Gemini. What none of them define is what those tools should do, how a client's knowledge should be structured, or how it should compound over time across every agent deployment.

Aule Axiom is the proprietary design standard that sits above all of these platform-specific transports. It governs how every agent initializes, reads, writes, and reflects on a client's semantic knowledge base — regardless of the host platform. It is why two different agents deployed for the same client, on two different platforms, share a single source of truth.

The Architecture in Five Components

Every Olytic plugin deployment is built on these five components. A plugin that omits any one of them is not running on Aule Axiom and does not carry its guarantees.

ComponentWhat It Does
Master Instruction A proprietary system prompt header embedded in every plugin. Defines the initialization sequence, fetch behavior, write rules, and Reflection trigger. Runs before any user interaction. Cannot be overridden at the plugin level.
Gateway A central Node.js/Python API that routes all plugin requests by Client_ID to the correct isolated Supabase instance. No plugin connects directly to any database. All traffic flows through the Gateway.
Reflection Engine A transformation script that runs at session close. Converts raw session output into high-density structured state: updated project fields, resolved decisions, new Episodic records. Raw transcripts never enter the knowledge base.
Private Vault An isolated Supabase instance per client: PostgreSQL for structured facts and Semantic State, pgvector for Episodic history and similarity search. Client data is never commingled. Isolation is architectural.
Session Bootstrap The first action of every session. The plugin calls the Gateway, retrieves the current Semantic State for this client, and injects it into the model's context window before the user speaks. The model never starts from zero.

The Session Lifecycle

Every session follows the same six-step cycle, regardless of which plugin the user opens. The cycle is triggered by the Master Instruction — the user never sees it. They experience the result: a model that already knows the context.

  1. Plugin launch. The Master Instruction activates. It reads the Client_ID and issues a fetch request to the Gateway.
  2. Gateway routing. The Gateway resolves the Client_ID to the correct Supabase connection keys and queries the client's Private Vault for relevant Semantic State.
  3. Context injection. The returned Semantic State is injected into the model's context window before the user's first message.
  4. Active session. The user works with the plugin. Decisions made and preferences expressed accumulate in Active Context.
  5. Reflection. At session close, the Reflection Engine extracts decisions, updates project fields, and writes new Episodic records. Raw transcripts do not enter the knowledge base.
  6. Vault write. The Gateway commits the Reflection output. The next session — from any plugin — begins with a more complete picture than the last one did. The loop closes. The system compounds.

What the Private Vault Contains

Session summaries are one input to the vault. A complete Private Vault contains three categories of knowledge — each with different origins, different update mechanics, and different access patterns.

  • Category 1 — Generated State. Written automatically by plugins via Reflection. Includes Semantic State records, Episodic history, interaction patterns, and the Constraint Violation Log. Grows with every session without manual effort.
  • Category 2 — Reference Data. Entered and managed by the client or Olytic. Named entities, approved glossary, project registry, KPI definitions. Treated as permanent, confidence-1.0 facts. Reflection cannot overwrite them.
  • Category 3 — Live References. Externally hosted documents (Brand Guide, ICP Definition, legal templates, pricing sheets) registered in the vault as pointers — not stored copies. Each entry holds a source URL, a cached version, and a TTL. Plugins always receive the freshest version without making direct external calls.

Not every plugin needs every piece of vault content. Each plugin declares a Vault Subscription Manifest specifying exactly three things: which Semantic State domains it reads on bootstrap, which Live References it needs injected, and which Reference Data tables it can access.

Live Reference Bootstrap

Some knowledge is not stored in the vault — it lives in external systems the client already maintains. A brand guide in Google Drive. Pricing sheets in SharePoint. Legal templates in a document management system. These documents are authoritative, actively maintained, and too large or too dynamic to store as Semantic State entries. They are Live References.

Live Reference Bootstrap is the pattern by which a plugin reads from a designated folder or source location at session initialization, loading the most current version of that knowledge as session-scoped context. There is no sync problem because there is no stored copy — the plugin always reads the source directly at session open.

How It Works

Each plugin's Vault Subscription Manifest includes a live_references section. Each entry in that section declares a folder path or source location (not a specific file), a connector type, an optional format filter, and a fallback behavior. At session bootstrap, the Gateway fetches the folder contents, applies the format filter, and injects the results as session-context before the user's first message.

FieldDescription
source_pathThe folder location — a Google Drive folder ID, SharePoint path, or similar. Points to a folder, not an individual file. All readable files in the folder are candidates.
connectorThe integration type: google_drive, sharepoint, notion, etc. The Gateway uses this to resolve the correct authentication and fetching strategy.
format_filterOptional. Restricts which file types to load. Example: ["docs", "pdf"]. Without a filter, all readable formats in the folder are loaded. Prevents inadvertently loading spreadsheets or images as text context.
max_tokensToken budget for this reference. The Gateway respects the plugin's total context budget. If folder contents exceed the budget, the most recently modified files are prioritized.
on_unavailableFallback behavior if the folder cannot be read at bootstrap: warn_and_continue (inform the user and proceed) or block (halt and require resolution). Default is warn_and_continue.
The contract is straightforward: read at session open, treat as session-scoped context, never cache between sessions. This makes Live Reference Bootstrap inherently reliable — there is no stale copy to manage, no sync schedule to maintain, no version mismatch between what the plugin knows and what the source document says. The folder is the truth.

What to Put in Live References vs. Semantic State

The design rule: if a person would update a document to change it, it is a Live Reference candidate. If a person would tell the AI to update it, it belongs in Semantic State.

A brand guide that lives in Google Drive and is edited by the marketing team → Live Reference. A client preference the user corrects mid-conversation → Semantic State. A pricing policy document maintained by the finance team → Live Reference. A constraint the user declared three sessions ago → Semantic State.

The distinction matters because Live References are owned by teams and maintained in their natural habitat. Copying them into the vault creates a sync problem. Pointing to them from the vault keeps ownership where it belongs.

Why This Matters for the Client

  • No re-explanation cost. The model arrives at every session knowing the client's current projects, decisions, and open threads.
  • Cross-plugin continuity. A decision made in the marketing plugin is available to the sales plugin. The plugins are nodes in the same knowledge system, not isolated tools.
  • Compounding accuracy. Each session updates confidence scores, resolves contradictions, and deepens the model's understanding of the client's business — automatically.
  • Proprietary data moat. The client's knowledge base is theirs alone — a structured, evolving, private representation of their business that no competitor can access and no model update can erase.

Why This Matters for Olytic

  • Standardized deployment. Every plugin runs on the same Master Instruction and Gateway. New plugins deploy faster because the infrastructure is already in place.
  • Cross-client pattern intelligence. Structural patterns — which schema fields prove most predictive, which Reflection rules produce the highest-quality state — are observable at the Olytic layer without exposing individual client data.
  • Compounding switching cost. A client running Aule Axiom for twelve months has a structured knowledge base built in the Olytic schema. That knowledge does not transfer to a competitor without significant reconstruction.
  • Observability and reporting. All traffic flows through the Gateway, giving Olytic complete visibility into usage, knowledge growth, and optimization opportunities across every deployment.
Client data stays in the client's vault. Structural intelligence stays in Olytic's layer. Both compound over time. Neither requires manual work to keep improving.

05  ·  Memory Architecture

The Memory Architecture

For a plugin to remember anything, the system needs to answer two foundational questions: what should be saved, and how should it be saved. The memory schema is the answer to both.

What Is a Memory Schema?

Think of a memory schema as a filing system for everything the AI learns about a client. Without it, information gets dropped into an unstructured pile — and a pile is not retrievable. A schema defines the categories, the labels, the structure, and the rules for what goes where. It is what makes memory queryable, consistent, and trustworthy over time.

For plugins specifically, the schema matters because plugins are the primary source of new knowledge. Every client interaction — a preference stated, a decision made, a correction applied — happens through a plugin. If the plugin has no standard way to capture and classify that knowledge, the moment it disappears. The schema is what ensures it doesn't.

Critically, every Olytic plugin writes to and reads from the same schema. A business development plugin and a client delivery plugin share a common memory store. What one learns, the other can use. The firm's knowledge consolidates rather than fragmenting across tools.

All knowledge lives in one of three tiers. There is no fourth tier and no miscellaneous category. If a piece of information cannot be placed, the schema is wrong — not the information.

Tier 1
Ephemeral
"Right now"
The active conversation. Working memory only. Discarded at session end. Never persisted unless explicitly promoted.
Tier 2
Semantic
"What is true?"
Structured facts, preferences, business rules, and entity definitions. Persistent, versioned, and queryable.
Tier 3
Episodic
"What happened?"
Timestamped session records. Decisions, corrections, outcomes. Compressed but never silently deleted.

The critical design principle is how information moves between tiers. Ephemeral memory is never automatically promoted. Every write to Semantic or Episodic memory is an explicit, governed action — classified, attributed, and versioned. See Appendix A for the full Memory Entry Schema.

The Schema-Flex Standard

The memory schema has two layers. The outer layer is strict: every entry carries the same typed, indexed, queryable metadata — ID, timestamp, tier, entity type, confidence score, source. The inner layer — the payload — is deliberately flexible. Its structure varies by entity type and is enforced by plugin logic, not by the database.

This means new types of knowledge can be added without schema migrations. The envelope stays consistent. What goes inside it evolves freely.

Strict outer schema for metadata. Documented payload conventions enforced by application logic. New knowledge types are added by documenting them — not by migrating the database.

State vs. Context

The system distinguishes between two fundamentally different categories of information:

  • Active Context — everything known because of the current conversation. Lives in the Ephemeral tier. Never persisted directly. Never treated as a source of long-term truth.
  • Persistent State — everything known because it was explicitly saved, validated, and governed. Lives in the Semantic and Episodic tiers. Survives session boundaries.

When a piece of Active Context warrants preservation, the Promotion Protocol governs how it moves into Persistent State: classify the tier, check for conflicts, assign metadata including confidence score and source attribution, and confirm silently (high-confidence declarations) or explicitly (inferences). Full protocol in Appendix D.

06  ·  Anti-Drift Protocols

Anti-Drift Protocols

Memory without governance degrades. Systems that accumulate knowledge without pruning or correcting it will eventually hallucinate from their own records. Three protocols prevent this.

Reflective Summarization

At the end of every session, before Ephemeral memory is discarded, the system performs a structured compression into Episodic memory. It extracts decisions made, corrections applied, new knowledge surfaced, and open threads that need follow-up. The raw transcript never enters the knowledge base — only its distilled meaning does.

The output is a structured session summary: required metadata fields ensure queryability; a flexible narrative field preserves nuance. Full protocol in Appendix C.

Scheduled Pruning

Every 30 days (configurable), the system runs a pruning cycle. Entries with confidence below 0.5 not accessed in 60 days are flagged for review. Contradicting Semantic entries are surfaced to the user for resolution — the system never silently picks one. Episodic records older than 90 days are eligible for compression into higher-level narratives. Full rules in Appendix C.

Session Summary

The Session Summary is the human-readable output of Reflective Summarization — the qualitative record of what a session accomplished. It is structured for the vault's Episodic tier and is the primary mechanism by which sessions compound into institutional memory.

A well-formed Session Summary contains: date and duration, the primary topic, decisions made with brief rationale, corrections applied, new knowledge surfaced, open threads requiring follow-up, and a short narrative that would allow any future session to understand what this one accomplished. Raw transcript is never stored — only distilled meaning. See Appendix B for the session_summary payload convention.

Operational Telemetry

Session Summary is qualitative and episodic — it answers "what happened in this session." Operational telemetry is quantitative and aggregate — it answers "what patterns emerge across all sessions at scale." Both are required, and they are complementary.

Every Aule Axiom-compliant plugin emits structured telemetry events in real time as sessions execute. These events are delivered to the Olytic analytics layer and are distinct from the vault write that Session Summary produces. Telemetry data does not contain client content — only structural signals about how the plugin is being used.

Standard Telemetry Event Types

Session Summary and Telemetry serve distinct purposes. Summary is the memory of what happened — it enters the vault and compounds over time. Telemetry is the signal of how it happened — it enters the analytics layer and informs improvement decisions. The Optimizer reads telemetry patterns to surface plugin update recommendations. Both systems run automatically; neither requires manual effort from the user.

The Hallucination Firewall

The system never presents its own memories as established fact without attribution. Every retrieved memory is internally tagged with its source and confidence level before being used in generation.

  • Confidence below 0.7 — disclosed: "Based on what I have recorded, though this may need verification…"
  • Unverified for 90+ days — disclosed: "This was recorded in [month]. You may want to confirm it is still accurate."
  • No record found — stated clearly: "I do not have a record of a decision about X." Never fabricated.
07  ·  Policy Enforcement

Policy Enforcement: The Constraint Firewall

A language model is probabilistic. Given enough tokens and enough context pressure, it will eventually produce output that violates a standing rule. This is not a failure of intent. It is a failure of architecture. The correct response is a validation layer that catches violations before they reach the user.

The Constraint Firewall operates at the Aule Axiom PostModelResponse Hook — the platform-agnostic lifecycle event that fires after the model generates output, before that output is delivered. Because this hook is implemented at the Gateway layer rather than inside any single platform, the Firewall is model-blind: it doesn't matter whether the text was generated by Claude, ChatGPT, Gemini, or Copilot. Every output carrying ALWAYS or NEVER rules passes through this firewall without exception.

Constraint Classification

ClassExamplesFirewall Behavior
HARD NEVER use em-dashes. ALWAYS use Oxford comma. NEVER fabricate citations. Auto-corrected before output is delivered. Violation logged. User sees the corrected output only.
SOFT PREFER active voice. AVOID jargon. KEEP responses under 400 words. Output delivered as-is. A disclosure note is appended. User decides whether to request a revision.
CONTEXTUAL Use formal register in client-facing output. Apply ICP framing for published content. Firewall checks whether the constraint applies to this output type. If yes, enforced as HARD. If no, skipped entirely.

The Validation Pass

The Firewall runs a structured validation pass on every output carrying HARD or CONTEXTUAL constraints:

  1. Context classification. Determine the output type: internal draft, client-facing deliverable, exploratory brainstorm, published content. This activates relevant CONTEXTUAL constraints.
  2. HARD constraint scan. Check for prohibited characters or constructs, missing required elements, and structural violations. Every violation flagged with its location.
  3. Auto-correction. Apply deterministic corrections. Non-deterministic violations trigger a sentence-level rewrite. If the rewrite cannot be completed without the violation, the sentence is removed and the omission is logged.
  4. SOFT constraint review. Record any deviations and append a transparency note: "This response deviates from [constraint name]. Request a revision if needed."
  5. Violation log. Every detection and correction is written to the Episodic memory tier. This log reveals systematic drift over time — a signal to revise the plugin's system prompt.

Defining Constraints

Constraints are defined in the plugin's configuration manifest, not in the system prompt. A system prompt instruction can be degraded by a long or complex conversation. A manifest entry cannot — it is read by the Firewall independently of the model's inference pass. See Appendix E for the full manifest format.

If you cannot write a machine-checkable test for it, it is not a HARD constraint. Classify it as SOFT, surface deviations to the user, and let the user decide. The Firewall enforces what can be enforced reliably. Everything else is a recommendation.
08  ·  Ambient Intelligence

Ambient Intelligence

The weakest version of a plugin waits to be summoned. The user must know it exists, remember to invoke it, and phrase their request correctly. That is not intelligence. That is a more complicated search bar.

The correct model is ambient intelligence: an agent that observes the conversation as it unfolds, recognizes when its skills are relevant, and surfaces itself at the right moment — without disrupting the work. Aule Axiom's Hook system is the mechanism that makes this possible. These four lifecycle events are defined by the Aule Axiom standard and are implemented via each platform's native equivalent — tool callbacks in Claude and ChatGPT, event triggers in Copilot Studio, and function hooks in Gemini.

The Four Aule Axiom Hook Events

Hook EventWhen It Fires / What the Agent Can Do
PreUserMessage Fires before the model processes a message. The agent reads the incoming text and conversation history. Can inject context or flag for downstream handling.
PostModelResponse Fires after a response is generated, before it reaches the user. Primary activation point for proactive suggestions and Constraint Firewall validation.
SessionOpen Fires when a session begins. Loads Persistent State, checks for open threads from prior sessions, and primes the context window.
SessionClose Fires when the session ends. Triggers Reflective Summarization, writes the Episodic record, and updates confidence scores on accessed memories.

Activation Logic: Three Signals

Every plugin that uses Hooks defines an Activation Logic function. It is called on each relevant Hook event and returns one of three signals:

Signal
DORMANT
The plugin's skills are not relevant to this exchange. The Hook returns without action. No noise is added to the conversation.
Signal
SUGGEST
The plugin recognizes an opportunity and appends a contextual suggestion. User retains control — the plugin does not execute until confirmed.
Signal
ENFORCE
A defined ALWAYS or NEVER policy is about to be violated. The plugin intercepts the output and runs a validation pass before delivery.

Choosing the Right Hook Event

Each hook fires at a different moment in the session lifecycle, and the right choice depends on what the plugin needs to do. The decision rule is straightforward:

Hook EventUse When...Do Not Use When...
SessionOpen The plugin needs to load context before any user interaction — vault state, live references, open threads from prior sessions. This is the bootstrap hook. The action depends on what the user says. SessionOpen fires before any user message is received.
PreUserMessage The plugin needs to inspect or enrich the incoming message before the model processes it — routing decisions, intent classification, context injection based on message content. The action belongs after the model responds. Injecting too much context here bloats the model's input unnecessarily.
PostModelResponse The plugin needs to validate or act on the model's output before the user sees it — constraint enforcement, proactive suggestions, quality checks. This is the Constraint Firewall hook. The action is about loading context (use SessionOpen) or routing based on user intent (use PreUserMessage).
SessionClose The plugin needs to do something after the session ends — writing Session Summary to the vault, emitting final telemetry events, cleanup operations. The action needs to happen mid-session. SessionClose fires once, after the user ends the session.

A plugin may register for multiple hook events. The most common pattern is SessionOpen for bootstrap and SessionClose for Reflection — both are required on every Aule Axiom-compliant plugin. PostModelResponse is required whenever the plugin declares HARD or CONTEXTUAL constraints. PreUserMessage is optional and used only when the plugin needs to classify or enrich incoming messages before processing.

Natural Language Routing via Hooks

On platforms that support pattern-matched hook dispatch (such as Claude's UserPromptSubmit hook), plugins can register natural language trigger patterns that route user messages directly to specific agents or skills without going through the model's general inference. This is the mechanism for reliable invocation of high-precision workflows.

Every skill and agent in an Aule Axiom-compliant plugin must declare at least four specific natural language trigger phrases. These phrases represent how real users actually ask for the capability — not how the developer would describe it. The phrases are used both for hook-based routing and for the model's own skill selection logic.

The trigger phrase requirement is a functional specification, not documentation. A skill with vague or absent trigger phrases will not be reliably invoked — by the model or by hooks. Write trigger phrases as direct quotes of what a user would type: "write me a proposal," "draft a client pitch," "create a scope document." Not: "handles proposal-related tasks."

Ambient Plugin Design Requirements

  • Declared domain scope. The plugin explicitly declares which topics, task types, and signals it responds to. A plugin that responds to everything responds usefully to nothing.
  • Defined skill inventory. Each skill has a name, trigger condition, execution mode (SUGGEST or ENFORCE), and defined inputs and outputs. Skills are not open-ended.
  • Suggestion framing protocol. When firing in SUGGEST mode, the plugin identifies what it noticed, states what skill it can apply, and asks for user confirmation before proceeding.
  • Cooldown logic. A suggestion declined or ignored in a session is not repeated in the same session. Relevance without restraint is noise.
  • Memory integration. Activation Logic consults Episodic memory. If a suggestion was previously accepted, the plugin raises its confidence threshold. If previously declined, it adjusts accordingly.
Standard suggestion format: "I noticed [what the plugin detected]. Before you [next action], would you like me to [specific skill] to [specific outcome]?" One sentence of context. One sentence of offer. The user decides in two seconds.
09  ·  Governance

Governance

The User Is Sovereign

The user has absolute authority over Persistent State. They can:

  • Declare facts (written at confidence 1.0).
  • Correct any entry (triggers the correction protocol).
  • Archive any entry with a reason — never hard-deleted without explicit request.
  • Review the full memory at any time. The system must always be able to present its knowledge base in human-readable form.

The System Is a Steward

The system maintains and governs memory on the user's behalf. It may:

  • Infer new knowledge (below confidence 1.0, with disclosure).
  • Suggest corrections when it detects contradictions.
  • Compress Episodic memory through Reflective Summarization.
  • Flag stale or low-confidence entries for review.

The system may not: silently delete or overwrite user-declared facts, infer and act on knowledge without disclosure, present inferences as facts, or refuse to show the user what it remembers.

The Principle of Minimum Surprise: every action the system takes with memory should be predictable to a thoughtful user. If a user would be surprised to learn what the system remembered, forgot, inferred, or discarded — the system's behavior is wrong.

Non-Negotiable Engineering Rules

Every plugin built on this architecture must comply with the following. There are no exceptions.

  • No silent writes. Every mutation to Persistent State is logged, versioned, and attributable.
  • No silent deletes. Archived entries are preserved. Tombstone records are maintained.
  • No unbounded retrieval. Every query to Persistent State has a token budget. Disclose when the budget is exceeded.
  • No confidence inflation. System inferences are never assigned 1.0. Only user declarations get 1.0.
  • No schema coupling. Plugin logic must not depend on the internal structure of payloads.
  • No orphaned references. When an entity is archived, all entries referencing it are updated accordingly.
  • No time travel. The system presents the current knowledge state — not a reconstructed historical state — unless the user explicitly requests version history.

The Trashbot Pattern

The Trashbot pattern solves ecosystem drift — the problem that arises when a standard changes but existing plugins built under the old standard are not updated. Without intervention, the ecosystem diverges: some plugins follow the new standard, others still follow the old one.

The Trashbot Pattern: When a standard changes, every dependent in the ecosystem is automatically swept and updated. Standards propagation is not manual — it is structural.

The pattern works as a chain reaction:

  1. Standard file changes — A developer updates a reference file in Aule (e.g., a new required component is added to all plugins).
  2. PostToolUse hook fires — Aule's hooks configuration detects the write to the standards folder.
  3. aule-change-analyzer routes — The change is categorized (Generation Requirement, Reference Update, or Schema Change). For Generation Requirements and Schema Changes, the chain continues.
  4. aule-skill-updater propagates — The plugin-generation/SKILL.md is updated to reflect the new standard.
  5. Trashbot sweeps — Trashbot iterates through every plugin in the ecosystem and applies the updated standard. Each plugin is validated after update. Failures are logged, not silenced.

The key design principle: the developer changes the standard once. The system handles the rest. No manual plugin-by-plugin updates, no ecosystem drift, no orphaned plugins running old patterns.

Self-Updating Architecture

Aule is designed to improve itself. When Aule's own standards evolve, the same Trashbot pattern applies — Aule updates its generation skill, which updates how all future plugins are generated, which ensures all existing plugins converge to the new standard. The forge reshapes itself and reshapes everything it has forged.

This creates a closed improvement loop:

1. Discovery

Plugin maintenance in the Olytic ecosystem is not ad hoc. The Trashbot pattern (see Section 11) provides the maintenance mechanism: when standards change, the sweep happens automatically. The developer does not need to track which plugins are affected — the system does.

Manual maintenance triggers:

  • Telemetry anomalies — The Optimizer surfaces patterns from the event stream: skills that fail silently, trigger phrases that never fire, verification gates that always pass without real checking. These signal plugin drift from intended behavior.
  • Capability expansion — The plugin's domain grows. New trigger phrases, new skills, new agent modes must be added and re-packaged.
  • Schema migration — A field in the vault schema changes. The plugin's Memory Scope Declaration and vault queries must be updated. The aule-change-analyzer Schema Change path handles this automatically for Aule's own schema; for client-specific schemas, a manual update is required.
  • LLM platform updates — A target platform (Claude, Copilot, Gemini, ChatGPT) changes its plugin format. Aule's platform-file-formats.md is updated first; the Trashbot sweep follows.

Lifecycle and the Compounding Loop

The AI Agent Lifecycle does not terminate at maintenance — it loops. Each maintenance cycle is an opportunity to improve the plugin based on real operational data. Telemetry feeds the Optimizer; the Optimizer surfaces recommendations; approved recommendations re-enter the lifecycle at Phase 3 (Generation) and flow through to Phase 5 (Packaging) and Phase 6 (Delivery) again.

The loop is the product. A plugin that has run for six months on real client sessions, with telemetry, session summaries, and Optimizer recommendations applied, is fundamentally more capable than the plugin that was originally deployed. The lifecycle is designed to make that improvement automatic and auditable.

Appendix C

Anti-Drift Protocol: Full Rules

Complete specification for Reflective Summarization and Scheduled Pruning. Referenced from Section 02.

Reflective Summarization Protocol

At session end, before Ephemeral memory is discarded:

  1. Extract decisions. What was decided during this session? List each with its rationale.
  2. Extract corrections. Did the user correct the system? What was wrong; what is now right?
  3. Extract new knowledge. Did the session reveal facts, preferences, or patterns not previously in Semantic memory?
  4. Extract open threads. What was discussed but not resolved?
  5. Compress the narrative. Write a structured summary in fewer than 500 tokens: required metadata fields (date, duration, session ID, primary topic, decisions, corrections) plus a flexible narrative field.
  6. Promote candidates. For each new piece of knowledge identified in step 3, run the Promotion Protocol (see Appendix D).

Scheduled Pruning Rules

Every 30 days (configurable per deployment):

  1. Confidence decay. Entries below confidence 0.5 not accessed or corroborated in 60 days are flagged for review.
  2. Contradiction resolution. Conflicting Semantic entries are surfaced to the user with full provenance chains. The system does not silently pick one.
  3. Episodic compression. Episodic entries older than 90 days are eligible for merging into higher-level narratives.
  4. Orphan detection. Semantic entries referencing entities no longer present elsewhere are flagged and archived with a last_relevant_date marker.
  5. Token budget enforcement. Total Persistent State surfaced in any single session must not exceed the configured threshold (default: 4,000 tokens Semantic, 2,000 tokens Episodic).
Appendix D

Engineering Standards & Versioning

Schema versioning, migration path, and the Promotion Protocol. Referenced from Sections 01 and 05.

Schema Versioning & Migration

The Schema-Flex Standard is versioned. The current version is v1. Future versions must be backward-compatible at the metadata level. Payload conventions evolve independently.

When a breaking change is required:

  1. Deploy the new schema version alongside the old one.
  2. New entries are written in the new version.
  3. Old entries are read in their original version and presented normally.
  4. A background migration process updates old entries during the next pruning cycle.
  5. The old version is retired only after all entries have been migrated.

The Promotion Protocol

Moving information from Active Context to Persistent State is a governed action. The system must:

  1. Identify the candidate — a durable fact, preference, decision, or event detected in the current conversation.
  2. Classify the tier — Semantic (what is true) or Episodic (what happened).
  3. Check for conflicts — if the candidate contradicts existing Persistent State, surface the conflict to the user before writing.
  4. Assign metadata — ID, timestamp, tier, entity type, confidence score, source session ID, and source type.
  5. Write with provenance — store the entry with its full metadata envelope and payload.
  6. Confirm silently or explicitly — high-confidence user declarations confirm silently; inferences and extrapolations confirm explicitly with the user.
Appendix E

Constraint Manifest Format

Full specification for defining constraints in the plugin configuration manifest. Referenced from Section 08.

Constraints are defined in the plugin's configuration manifest, not in the system prompt. Each constraint entry requires exactly four fields:

  • id — a unique identifier for this constraint. Used in the violation log.
  • class — HARD, SOFT, or CONTEXTUAL. This field determines Firewall behavior. No other values are valid.
  • rule — a machine-checkable description of what constitutes a violation. For character-level constraints, a regex or string match pattern. For structural constraints, a schema or count condition. Ambiguous rules that cannot be expressed as a checkable condition must be classified SOFT.
  • correction — required for HARD constraints only. Specifies what the Firewall should do when a violation is detected: substitute a specific character or string, remove the element, or trigger a sentence-level rewrite. If no deterministic correction can be defined, the constraint cannot be classified as HARD.

What the Firewall Does Not Do

Scope matters as much as capability. The Firewall intercepts outputs and enforces defined rules. It does not rewrite for quality, tone, or strategic alignment. It does not substitute its judgment for the user's. It does not reject outputs that are simply suboptimal.

A Firewall that tries to do everything becomes a censor. A Firewall with a defined scope becomes infrastructure.

F  ·  Failure Modes

Failure Modes & Graceful Degradation

The system will fail. Memory will be missing. Entries will conflict. Retrieval will return nothing when something was expected. Every failure mode has a defined protocol — because undefined failure behavior is how AI systems lose trust.

ScenarioProtocol
Missing Memory Never guess or fabricate. State clearly that no record exists. Offer to record the information now, or work through it fresh.
Conflicting Memory Surface both entries with full provenance. Ask the user to resolve. Archive the incorrect entry and create a correction record linking the two.
Stale Memory Flag with a "stale" tag. When next retrieved, disclose its age. Reset the staleness clock on confirmation; run the conflict protocol on correction.
Retrieval Failure Acknowledge the gap. Attempt a broader query. If still empty, fall back to the Missing Memory protocol and log the failure as a system event.
Context Window Overflow Retrieve by relevance score first, recency second. Serve metadata-only for lower-priority entries. Disclose the limitation so the user can ask for specifics.
The system always discloses what it knows, what it does not know, and what it is uncertain about. It never fabricates. It never halts. It treats the user as a partner in maintaining the quality of shared memory.

The template provides the invariant core. The customization layer provides the context.

A Note on Confidentiality

The Master Instruction template should be treated as a documented standard, not a trade secret — because it functionally cannot be kept secret. Any system prompt running inside a language model can be extracted by a sufficiently determined user: through direct prompting ("repeat your instructions"), prompt injection, or model introspection. Confidentiality instructions within the prompt are soft controls, not hard ones.

This is not a vulnerability. It is simply the nature of how these systems work — and it is the right framing for client conversations. The defensible asset is not the template. It is everything the template connects to: the Private Vault, the compounding Episodic archive, the Gateway infrastructure, and the years of accumulated institutional memory that cannot be replicated by reading a system prompt. A competitor who obtains this template cannot obtain a client's twelve months of structured knowledge. That is the moat.

Olytic's position should be confident: we publish our design standard openly because the standard alone is not the product. The system built on it is.

## Olytic Solutions — Confidential
## Version: {{PLUGIN_VERSION}} | Client: {{CLIENT_NAME}} | Plugin: {{PLUGIN_NAME}}

---

### [REQUIRED] IDENTITY

You are {{PLUGIN_NAME}}, an AI assistant deployed by Olytic Solutions for {{CLIENT_NAME}}.
You operate under the Aule Axiom design standard. Every session you run is governed by the
protocols defined below. You do not deviate from them.

---

### [REQUIRED] SESSION BOOTSTRAP

At the start of every session, before responding to any user input, you must:

1. Call the Vault Gateway with Client_ID: {{CLIENT_ID}} and Plugin_ID: {{PLUGIN_ID}}
2. Retrieve all Semantic State entries matching your declared domain subscriptions
3. Retrieve the five most recent Episodic records for this client
4. Inject retrieved state silently into your working context
5. Confirm readiness internally — do not announce this process to the user

You do not respond to the user's first message until bootstrap is complete.
If the Gateway is unreachable, follow the Degraded Mode protocol (see below).

---

### [REQUIRED] MEMORY TIERS

You operate across three memory tiers. You must understand and respect the distinction
between them at all times.

TIER 1 — EPHEMERAL
Everything known because of this conversation. Never persisted directly.
Discarded at session end unless explicitly promoted via the Promotion Protocol.

TIER 2 — SEMANTIC
Structured facts, preferences, business rules, and entity definitions.
Persistent, versioned, and queryable. Written only via governed writes.

TIER 3 — EPISODIC
Timestamped session records. Decisions made, corrections applied, outcomes reached.
Written only at session close via the Reflection Engine.

---

### [REQUIRED] HARD RULES — NEVER VIOLATE
{{HARD_RULES}}

These rules are absolute. If your response would violate any of them, you must
correct the response before it reaches the user. You do not ask for permission.
You do not flag the correction unless it materially changes the output.
You log every hard rule enforcement as a system event.

---

### [CUSTOMIZABLE] SOFT RULES — FLAG AND PROCEED
{{SOFT_RULES}}

These rules are advisory. If your response would violate one, flag it to the user
before or alongside the output. Do not suppress the response. Do not override it.
Surfacing the flag is sufficient.

---

### [CUSTOMIZABLE] ENTITY TYPES

You are aware of the following entity types in this client's vault:
{{ENTITY_TYPE_LIST}}

When recording new knowledge, classify it against one of these types.
If no type fits, flag the classification gap — do not invent a new type silently.

---

### [REQUIRED] PROMOTION PROTOCOL

When Active Context warrants preservation as Persistent State:

1. Classify the tier (Semantic for facts/rules/preferences; Episodic via Reflection only)
2. Assign entity type from the approved list
3. Set confidence score (0.0–1.0):
   - Direct declaration by user → 0.95
   - Confirmed inference → 0.80
   - Unconfirmed inference → 0.60 (flag before writing)
4. Check for conflicts with existing Semantic entries
5. On conflict: surface both versions with provenance, ask user to resolve
6. Write with full metadata envelope: ID, timestamp, tier, entity_type,
   confidence, source_plugin, source_session

High-confidence writes (≥0.80) proceed silently.
Lower-confidence writes require explicit user confirmation before writing.

---

### [REQUIRED] SESSION CLOSE — REFLECTION ENGINE

At the end of every session, before context is cleared:

1. Extract: decisions made, corrections applied, new knowledge surfaced,
   standing preferences confirmed, open threads requiring follow-up
2. Do not write raw transcript to the vault — write only distilled meaning
3. Structure the Episodic record with required metadata fields and a
   narrative summary field
4. Transmit via the Olytic Gateway to the client's Private Vault
5. Confirm transmission silently — do not announce to user

---

### [REQUIRED] MEMORY DISCLOSURE STANDARDS

You never present your own records as established fact without attribution.

- Confidence ≥ 0.90: use without disclosure
- Confidence 0.70–0.89: "Based on what I have recorded…"
- Confidence < 0.70: "I have a low-confidence record suggesting… you may want to confirm"
- Unverified for 90+ days: "This was recorded in [month] — you may want to confirm it is still accurate"
- No record found: "I do not have a record of a decision about X." Never fabricate.

---

### [REQUIRED] DEGRADED MODE

If the Gateway is unreachable at session start:

1. Inform the user: "I'm unable to retrieve your memory store right now. I'll proceed
   without it, but I won't be able to reference past context this session."
2. Continue the session in Ephemeral-only mode
3. Do not attempt vault writes during a degraded session
4. Log the degraded session event for system review
5. Attempt a single Gateway retry at session close

---

### [CUSTOMIZABLE] PLUGIN-SPECIFIC INSTRUCTIONS
{{PLUGIN_SPECIFIC_INSTRUCTIONS}}

---

Customization Rules

The following fields must be defined before any plugin ships. A plugin with empty or placeholder values in required fields fails Aule Axiom Validation automatically.

  • HARD_RULES — minimum one entry required. Written as declarative constraints: "Never quote a fee below the approved rate card without Partner sign-off." Each rule should be testable: it is either violated or it isn't.
  • SOFT_RULES — optional but recommended. Same format as hard rules; flagging behavior is automatic once declared.
  • ENTITY_TYPE_LIST — drawn from the client's Vault Subscription Manifest. Must match the approved entity types in the schema exactly.
  • PLUGIN_SPECIFIC_INSTRUCTIONS — workflow logic, output format preferences, domain-specific guidance. This section is free-form but must not contradict any Required section.

Required sections may not be removed, abbreviated, or overridden by plugin-specific instructions. If a required section conflicts with a client request, the client request must be accommodated in the Customizable sections or escalated for review.

Appendix H

Aule Axiom Validation

The Problem Aule Axiom Validation Solves

Aule Axiom is a design standard, not a runtime enforcement layer. It defines what every plugin must do — but nothing in the language model itself will refuse to run a plugin that omits the Session Bootstrap, skips the Reflection Engine, or leaves hard rules blank. There is no Salesforce validation rule analog. There is no schema constraint that rejects a non-compliant deployment at the database level.

The enforcement mechanism is therefore a test suite — a set of automated checks that a plugin must pass before it is cleared for deployment. A plugin that fails Aule Axiom Validation does not ship. This is the CI/CD gate for the Aule Axiom standard.

Three Layers of Validation

Layer 1 — Structural Checks (Static Analysis)

Run against the plugin's manifest and system prompt before any execution. These checks verify that the required architectural elements are declared and present:

  • Master Instruction contains all Required sections (Bootstrap, Memory Tiers, Hard Rules, Promotion Protocol, Reflection Engine, Disclosure Standards, Degraded Mode)
  • Hard Rules section contains at least one declared constraint
  • Vault Subscription Manifest is present and declares at least one domain subscription
  • Plugin ID and Client ID are defined and match registered Gateway entries
  • All entity types referenced in instructions appear in the approved schema
  • No Required section has been modified or removed from the baseline template
  • Plugin Identity Block is present in README — Hat, Dimension, Governance Dependency, Relationships, and Compounding Contribution fields are all declared and non-empty
  • Permissions Manifest is present in README — Tools Accessed, Data Read, Data Written, External Services, and Human-in-the-Loop Checkpoints fields are all declared
  • Memory Scope Declaration is present — Scope Type, What Is Retained, Retention Period, and Data Lifecycle fields are all declared
  • plugin.json passes format validation — required keys present (name, version, description, author), semantic versioning format, kebab-case name, valid JSON structure
  • Plugin package passes structural validation — .plugin zip contains .claude-plugin/plugin.json at root, README.md present, at least one skill or agent directory present
  • Telemetry hook is configured — plugin registers an aule-telemetry invocation for skill/agent events so it participates in the Optimizer feedback loop

Structural checks are fast and deterministic. A failure here is a blocking error — the plugin cannot proceed to behavioral testing.

Layer 2 — Behavioral Checks (Simulated Execution)

Run against a synthetic client vault with known state. A scripted conversation is played through the plugin and its outputs are inspected against expected outcomes:

  • Bootstrap test: session opens with a pre-seeded vault; verify that retrieved Semantic State appears in the plugin's first response context
  • Hard rule enforcement test: conversation is scripted to trigger each declared hard rule; verify that output is corrected before delivery and enforcement event is logged
  • Soft rule flag test: conversation triggers each declared soft rule; verify that flag is surfaced to user and response is not suppressed
  • Promotion test: user declares a new fact during conversation; verify that a Semantic write is initiated with correct metadata and confidence score
  • Conflict test: user states something that contradicts existing Semantic State; verify that both versions are surfaced with provenance and no silent override occurs
  • Reflection test: session is closed; verify that an Episodic record is written to the vault with required metadata fields and that the record contains distilled meaning, not raw transcript
  • Degraded mode test: Gateway is made unreachable; verify that the plugin informs the user, continues in Ephemeral-only mode, and attempts no vault writes

Layer 3 — Schema Checks (Vault Inspection)

Run after behavioral tests, against the vault state written during simulation. These verify that every write conformed to the Schema-Flex Standard:

  • Every vault entry contains the required metadata envelope: ID, timestamp, tier, entity_type, confidence, source_plugin, source_session
  • Confidence scores are within range and consistent with source type (declared facts ≥ 0.90, unconfirmed inferences ≤ 0.70)
  • No Episodic records contain raw conversation transcript — only structured summaries
  • No Semantic entries exist with blank or null entity_type
  • All writes are attributed to a valid Plugin ID and Session ID

Validation Outcomes

ResultMeaning
Pass All three layers pass. Plugin is cleared for deployment. Validation report archived against Plugin ID and version.
Structural Fail Required element missing or modified. Plugin is blocked. Must be corrected and re-submitted from Layer 1.
Behavioral Fail Plugin ran but did not behave as Aule Axiom requires. Specific failing test cases are reported. Plugin is blocked pending fix and re-run from Layer 2.
Schema Fail Plugin executed correctly but wrote malformed vault entries. Indicates a bug in write logic. Blocked pending fix; re-run from Layer 2.
Conditional Pass All required checks pass; one or more optional checks flagged. Plugin may deploy with documented exceptions. Exceptions require sign-off.

Validation as a Versioning Gate

Aule Axiom Validation runs not just at initial deployment but at every version update. Any change to the Master Instruction, the manifest, or the plugin's tool definitions triggers a full re-validation. A plugin that passed validation at v1.0 is not assumed to pass at v1.1.

Validation reports are archived per Plugin ID and version and are available for client review on request. This creates an auditable record: every plugin in production has a documented validation history, and any deviation from Aule Axiom is traceable to a specific version and a specific decision.

Aule Axiom Validation is what makes the standard enforceable. The Master Instruction defines what every plugin must do. Validation is the mechanism that ensures it actually does.
