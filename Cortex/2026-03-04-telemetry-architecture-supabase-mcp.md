# Telemetry Architecture: Org-Scoped Supabase MCP Connector

**Date:** 2026-03-04
**Status:** Ready for processor

## The Problem

Plugin telemetry needs to reach Supabase from any Cowork session, for any user, with zero manual setup. Three approaches were attempted and failed:

1. **Bash file writes to `~/.claude/telemetry/`** — Claude's Bash tool runs in a Linux VM sandbox. Writes go to the VM filesystem, which resets between sessions. Files never accumulated on the user's Mac.
2. **Supabase Edge Function via HTTP MCP server (`.mcp.json` in plugin zip)** — Cowork does not register MCP servers declared in plugin zip `.mcp.json` files. The `olytic-telemetry` tool was never available in sessions.
3. **WebFetch / curl to Edge Function** — Egress is blocked by Cowork's network proxy. Neither WebFetch nor Bash curl can reach external URLs including Supabase.

## The Insight

The correct architecture uses the **org-scoped Supabase MCP connector** (`execute_sql`) that is installed at the org level in Cowork — not an HTTP MCP server, not file writes, not outbound HTTP from the VM.

**How it works:**
- Each org gets a Supabase MCP connector provisioned by Olytic, pointed at Olytic's `telemetry_events` table, credentialed with a key Olytic controls
- The connector is installed once at the org level — it's available in every session for every plugin that org uses
- Plugin telemetry skills identify the connector at runtime by looking for the tool matching the pattern `mcp__[connector-id]__execute_sql`
- Skills call `execute_sql` with a direct INSERT statement — no Edge Function, no staging files, no user setup

**Confirmed working:** `execute_sql` successfully inserted rows into `telemetry_events` in a live test. Row returned with ID. This is the only outbound data path that works reliably in the Cowork environment.

## Why It Matters

- **Zero user setup** — the connector is provisioned by Olytic as part of org onboarding, not by the client
- **Works for all plugins** — any plugin Olytic deploys for that org inherits the same connector
- **Real-time** — events land in Supabase immediately, no batching or transmission delay
- **Scales to multi-org** — each org gets their own connector entry; all data flows into one Olytic-controlled Supabase instance
- **No infrastructure to maintain** — no Edge Function needed, no startup scripts, no hooks

## Relevant Plugins

- Gaudi (owns the telemetry architecture decision)
- Aule (generates plugin-telemetry skills for new plugins — must use this pattern in the template)
- All four existing plugins (gaudi, aule, magneto, the-one-ring — all updated to use execute_sql)

## Suggested Implementation

**For existing plugins:** Plugin-telemetry skill's "How to Write Logs" section instructs Claude to:
1. Identify the Supabase MCP connector in the session (tool pattern: `mcp__[connector-id]__execute_sql`)
2. Call `execute_sql` with the Olytic telemetry project ID and an INSERT statement
3. Only include columns with values — omit optional fields rather than inserting nulls
4. Never display the result to the user

**For new client plugins generated by Aule:** The telemetry-template.md has been updated to use the same `execute_sql` pattern. Aule will bake this into every generated plugin automatically.

**For org onboarding:** When Olytic sets up a new client org, provision a Supabase MCP connector as part of their Cowork configuration. This is a one-time step per org, handled by Olytic — not the client.

**What NOT to do (for future reference):**
- Do not attempt to write telemetry to local file paths — the VM sandbox prevents this from working
- Do not use WebFetch or curl to call external APIs — egress is blocked
- Do not rely on `.mcp.json` in plugin zips to register MCP servers — Cowork does not load these
