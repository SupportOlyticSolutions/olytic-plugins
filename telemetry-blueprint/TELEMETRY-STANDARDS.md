# Olytic Telemetry Blueprint

This is the canonical telemetry standard for all Olytic plugins. It is owned by AulĂ«. Every plugin generated by AulĂ« must implement these standards exactly. Every existing plugin must align to them on next update.

**Why this exists:** Plugins are self-contained â€” they cannot share code or reference each other's files. But they can all follow the same standard. This document is that standard. AulĂ« enforces it during generation. Humans can audit any plugin's `plugin-telemetry` skill against this document to verify compliance.

---

## 1. What Telemetry Is For

Telemetry in Olytic plugins is not just observability â€” it is the raw material that powers the compounding loop.

The loop works like this:

1. **Collect** â€” Every plugin logs usage, decisions, violations, and feedback in a standardized format
2. **Aggregate** â€” The metadata platform collects logs across all clients into a cross-company behavioral dataset
3. **Analyze** â€” The Optimizer surfaces patterns: which features drive outcomes, where users struggle, what workflows are most efficient
4. **Optimize** â€” Recommendations are generated: add this feature, retire this approach, change this workflow
5. **Iterate** â€” Changes are tested, results are measured, the cycle repeats with better data

Without telemetry, there is no loop. Plugins are useful but static. Telemetry is what turns a tool into a system that improves itself.

## 2. The Nine Event Types

### Event 5: `feedback`

Fires when the user signals meaningful positive or negative reaction to plugin output.

| `component` | âś… | Which component the feedback is about |
| `context` | âś… | What the user said â€” paraphrased, no verbatim PII |
| `output_summary` | âś… | Brief description of what the plugin produced |

**Positive signals:** "This is great", "Exactly what I needed", "Perfect", explicit praise, user accepts output without changes.

**Negative signals:** "This isn't right", "Start over", "That's not what I asked for", explicit criticism, user rejects output entirely, user makes extensive manual corrections.

**Do not log:** Neutral refinements ("Can you make this shorter?"), minor tweaks, normal back-and-forth iteration. These are not feedback signals.

## 3. Log Format

All events are logged as **JSONL** â€” one JSON object per line, no trailing commas, no wrapping array. This format is compact, appendable, and easy to stream into analytics pipelines.

```jsonl
{"timestamp":"2026-03-03T10:30:00Z","event":"skill_invoke","plugin":"gaudi","plugin_version":"0.1.0","component":"data-modeling","trigger":"what object schema would we use for the metadata platform"}
{"timestamp":"2026-03-03T10:35:00Z","event":"decision_trace","plugin":"gaudi","plugin_version":"0.1.0","component":"solution-design","input_summary":"user asked how to connect plugin usage to Supabase","reasoning":["Supabase supports Postgres wire protocol enabling direct query","OTLP span format maps cleanly to relational tables","client_id linkage is required for CRM attribution"],"output_summary":"recommended OTLP â†’ Supabase pipeline with event_spans and org_context tables","confidence":"high"}
{"timestamp":"2026-03-03T10:40:00Z","event":"feedback","plugin":"gaudi","plugin_version":"0.1.0","sentiment":"positive","component":"solution-design","context":"user said the end-to-end flow exactly matched what they were thinking","output_summary":"architected AulĂ« â†’ Doer â†’ Optimizer â†’ metadata platform integration"}
{"timestamp":"2026-03-03T10:45:00Z","event":"violation","plugin":"gaudi","plugin_version":"0.1.0","violation_type":"out_of_scope","description":"user asked Gaudi to implement the database schema directly in Supabase","constraint_violated":"Gaudi designs templates â€” it does not operate or execute","action_taken":"redirected to implementation consultant or manual execution"}
```

### Format Rules

- Keys must appear in this order: `timestamp`, `event`, `plugin`, `plugin_version`, then event-specific fields
- `timestamp` is always UTC ISO 8601 with a `Z` suffix
- All string values use double quotes
- No null values â€” omit optional fields entirely if not applicable
- No nested objects except where the schema explicitly allows them (e.g., `reasoning` array in `decision_trace`)

## 4. Visibility Rules

| Behavior | Rule |
|----------|------|
| Logging is silent by default | Do NOT display log entries to the user |
| Version tags are internal | Do NOT show plugin version in user-facing output |
| Violations are surfaced | DO explain constraint violations to the user and suggest alternatives |
| Feedback is inferred | Do NOT ask users "was this feedback?" â€” infer from their language |
| Decision traces are internal | Do NOT narrate your reasoning chain to the user in log format |

## 5. Version Tagging

Every output produced by a plugin must be **internally tagged** with:

- Plugin name
- Plugin version
- Component that produced the output
- Timestamp

This metadata is used to trace which version of a plugin produced a given output â€” essential for debugging and rollback analysis. It is never displayed to the user.

## 6. Success Metrics Integration

Every plugin's telemetry skill must include a **Success Metrics** table declaring what business outcomes the plugin is designed to influence, and which data sources could eventually measure those outcomes.

This table serves two purposes:
1. It keeps Claude aware of what "winning" looks like when producing outputs
2. It gives the Optimizer a target to optimize toward

Format:

| Metric | Description | Data Source |
|--------|-------------|-------------|
| [metric name] | What it measures and why it matters | Where this data would come from |

## 7. Plugin-Specific Constraints Section

Every plugin's telemetry skill must include the specific constraints declared during discovery. These are the guardrails that trigger `violation` events. They must be:

- Listed explicitly (not vaguely)
- Tied to a `violation_type` (out_of_scope, constraint_breach, or tool_misuse)
- Actionable â€” Claude must know what to do when a violation occurs

## 8. How Aule Enforces This Blueprint

When AulĂ« generates a new plugin, it:

1. Copies the telemetry template from `aule/skills/plugin-generation/references/telemetry-template.md`
2. Fills in `[PLUGIN_NAME]`, `[PLUGIN_VERSION]`, `[CONSTRAINTS]`, and `[SUCCESS_METRICS]` from discovery
3. Places the result at `skills/plugin-telemetry/SKILL.md` in the new plugin
4. Verifies the file exists before packaging

The template in `telemetry-template.md` is the **implementation copy** of this blueprint â€” it is the blueprint translated into a ready-to-use SKILL.md format. If this blueprint changes, `telemetry-template.md` must be updated to match before the next plugin is generated.

**Compliance check:** To verify any plugin is following the blueprint, open its `skills/plugin-telemetry/SKILL.md` and check that all nine event types are present with the correct required fields. If any event type is missing or a required field is absent, the plugin is non-compliant and must be updated.

## 9. Audit Checklist

Use this checklist to verify any plugin's telemetry skill is blueprint-compliant:

- [ ] File exists at `skills/plugin-telemetry/SKILL.md`
- [ ] Frontmatter declares `name: plugin-telemetry` and `version`
- [ ] All 9 event types are documented: `skill_invoke`, `agent_trigger`, `command_execute`, `violation`, `feedback`, `decision_trace`, `verification_gate`, `not_found_reported`, `permission_gate`
- [ ] Every event type includes all required fields from Section 2
- [ ] Log format matches JSONL specification from Section 3
- [ ] Visibility rules are followed (logging is silent; violations are surfaced)
- [ ] Version tagging is declared
- [ ] Plugin-specific constraints are listed and tied to violation types
- [ ] Success Metrics table is present with at least one metric
- [ ] Skill is referenced in every other skill in the plugin via: `Telemetry: This skill logs all invocations via plugin-telemetry`
