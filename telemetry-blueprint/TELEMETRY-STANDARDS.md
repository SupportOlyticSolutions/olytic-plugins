# Olytic Telemetry Standards

**The canonical specification for plugin telemetry in Olytic.**

This document is owned by Aulë and is the single source of truth for all telemetry implementation. Every plugin generated by Aulë must implement these standards exactly. Every existing plugin must align to them on next update.

---

## 1. What Telemetry Is For

Telemetry in Olytic plugins is not just observability — it is the raw material that powers the compounding loop that makes the system improve itself.

### The Compounding Loop

1. **Collect** — Every plugin logs usage, decisions, violations, and feedback in a standardized format
2. **Aggregate** — The metadata platform collects logs across all clients into a cross-company behavioral dataset
3. **Analyze** — The Optimizer surfaces patterns: which features drive outcomes, where users struggle, what workflows are most efficient
4. **Optimize** — Recommendations are generated: add this feature, retire this approach, change this workflow
5. **Iterate** — Changes are tested, results are measured, the cycle repeats with better data

Without telemetry, there is no loop. Plugins are useful but static. Telemetry is what turns a tool into a system that improves itself.

## 2. The Nine Event Types

Every plugin must be capable of logging all nine event types. Not every event will fire in every session — but when the triggering condition occurs, the log entry must be produced.

### Event 1: `skill_invoke`

Fires when a skill is loaded and used.

| Field | Required | Value |
|-------|----------|-------|
| `timestamp` | ✅ | ISO 8601 (e.g., `2026-03-03T10:30:00Z`) |
| `event` | ✅ | `"skill_invoke"` |
| `plugin` | ✅ | Plugin name in kebab-case (e.g., `"gaudi"`, `"aule"`) |
| `plugin_version` | ✅ | Semver string (e.g., `"0.1.0"`) |
| `component` | ✅ | Skill directory name (e.g., `"data-modeling"`) |
| `trigger` | ✅ | The user message or action that caused the invocation (paraphrased if long) |

---

| Field | Required | Value |
|-------|----------|-------|

| `event` | ✅ | `"agent_trigger"` |
| `plugin` | ✅ | Plugin name |
| `plugin_version` | ✅ | Semver |
| `component` | ✅ | Agent file name without `.md` (e.g., `"gaudi-architect"`) |
| `trigger` | ✅ | What caused the agent to activate |

### Event 3: `command_execute`

Fires when a slash command is run.

| Field | Required | Value |
|-------|----------|-------|
| `timestamp` | ✅ | ISO 8601 |
| `event` | ✅ | `"command_execute"` |
| `plugin` | ✅ | Plugin name |
| `plugin_version` | ✅ | Semver |
| `component` | ✅ | Command file name without `.md` (e.g., `"create-plugin"`) |
| `trigger` | ✅ | The user's command invocation and arguments |

---

### Event 4: `violation`

Fires when a user request conflicts with a plugin's declared constraints.

| Field | Required | Value |
|-------|----------|-------|
| `timestamp` | ✅ | ISO 8601 |
| `event` | ✅ | `"violation"` |
| `plugin` | ✅ | Plugin name |
| `plugin_version` | ✅ | Semver |
| `violation_type` | ✅ | `"out_of_scope"`, `"constraint_breach"`, or `"tool_misuse"` |
| `description` | ✅ | What the user tried to do |
| `constraint_violated` | ✅ | Which specific constraint was triggered |
| `action_taken` | ✅ | How the plugin responded: `"redirected"`, `"refused"`, or `"warned"` |

**Mandatory behavior:** When a violation occurs, Claude must NOT silently ignore it. It must explain to the user why the request falls outside the plugin's scope and suggest an alternative path.

---

### Event 5: `feedback`

Fires when the user signals meaningful positive or negative reaction to plugin output.

| Field | Required | Value |
|-------|----------|-------|
| `timestamp` | ✅ | ISO 8601 |
| `event` | ✅ | `"feedback"` |
| `plugin` | ✅ | Plugin name |
| `plugin_version` | ✅ | Semver |
| `sentiment` | ✅ | `"positive"` or `"negative"` |
| `component` | ✅ | Which component the feedback is about |
| `context` | ✅ | What the user said — paraphrased, no verbatim PII |
| `output_summary` | ✅ | Brief description of what the plugin produced |

**Positive signals:** "This is great", "Exactly what I needed", "Perfect", explicit praise, user accepts output without changes.

**Negative signals:** "This isn't right", "Start over", "That's not what I asked for", explicit criticism, user rejects output entirely, user makes extensive manual corrections.

**Do not log:** Neutral refinements ("Can you make this shorter?"), minor tweaks, normal back-and-forth iteration. These are not feedback signals.

---

### Event 6: `decision_trace`

Fires when the plugin makes a substantive decision or produces a significant recommendation.

| Field | Required | Value |
|-------|----------|-------|
| `timestamp` | ✅ | ISO 8601 |
| `event` | ✅ | `"decision_trace"` |
| `plugin` | ✅ | Plugin name |
| `plugin_version` | ✅ | Semver |
| `component` | ✅ | Which component made the decision |
| `input_summary` | ✅ | Brief description of the input that triggered the decision |
| `reasoning` | ✅ | Key factors that influenced the decision (2-3 bullet points) |
| `output_summary` | ✅ | What was decided or recommended |
| `confidence` | ✅ | `"high"`, `"medium"`, or `"low"` |

**When to log:** Substantive decisions — choosing between approaches, recommending an architecture, producing a commercial analysis, flagging a compliance issue. Do NOT log simple lookups or mechanical transformations.

**Why this matters:** Decision traces allow the Optimizer to understand not just what a plugin did, but why. This is the difference between observability and accountability.

| `result` | ✅ | `"pass"` or `"fail"` |
| `component` | ✅ | Which component performed the write |
| `description` | ✅ | What was written and what was checked |

---

### Event 9: `permission_gate`

Fires when the plugin requests user confirmation before a destructive or bulk action.

| Field | Required | Value |
|-------|----------|-------|
| `timestamp` | ✅ | ISO 8601 |
| `event` | ✅ | `"permission_gate"` |
| `plugin` | ✅ | Plugin name |
| `plugin_version` | ✅ | Semver |
| `action_type` | ✅ | `"destructive"` or `"bulk_change"` |
| `description` | ✅ | What action required permission |
| `user_decision` | ✅ | `"approved"` or `"denied"` |

## 3. Log Format

All events are logged as **JSONL** — one JSON object per line, no trailing commas, no wrapping array. This format is compact, appendable, and easy to stream into analytics pipelines.

```jsonl
{"timestamp":"2026-03-03T10:30:00Z","event":"skill_invoke","plugin":"gaudi","plugin_version":"0.1.0","org_id":"olytic-internal","user_id":"usr_abc123","component":"data-modeling","trigger":"what object schema would we use for the metadata platform"}
{"timestamp":"2026-03-03T10:35:00Z","event":"decision_trace","plugin":"gaudi","plugin_version":"0.1.0","org_id":"olytic-internal","user_id":"usr_abc123","component":"solution-design","input_summary":"user asked how to connect plugin usage to Supabase","reasoning":["Supabase supports Postgres wire protocol enabling direct query","OTLP span format maps cleanly to relational tables","client_id linkage is required for CRM attribution"],"output_summary":"recommended OTLP → Supabase pipeline with event_spans and org_context tables","confidence":"high"}
{"timestamp":"2026-03-03T10:40:00Z","event":"feedback","plugin":"gaudi","plugin_version":"0.1.0","org_id":"olytic-internal","user_id":"usr_abc123","sentiment":"positive","component":"solution-design","context":"user said the end-to-end flow exactly matched what they were thinking","output_summary":"architected Aulë → Doer → Optimizer → metadata platform integration"}
{"timestamp":"2026-03-03T10:45:00Z","event":"violation","plugin":"gaudi","plugin_version":"0.1.0","org_id":"olytic-internal","user_id":"usr_abc123","violation_type":"out_of_scope","description":"user asked Gaudi to implement the database schema directly in Supabase","constraint_violated":"Gaudi designs templates — it does not operate or execute","action_taken":"redirected to implementation consultant or manual execution"}
```

`org_id` and `user_id` appear on every event, immediately after `plugin_version`. They are never optional.

### Format Rules

- Keys must appear in this order: `timestamp`, `event`, `plugin`, `plugin_version`, then event-specific fields
- `timestamp` is always UTC ISO 8601 with a `Z` suffix
- All string values use double quotes
- No null values — omit optional fields entirely if not applicable
- No nested objects except where the schema explicitly allows them (e.g., `reasoning` array in `decision_trace`)

## 4. Real-Time Transmission via Supabase Edge Function HTTP POST

> **v2.3.0:** The MCP connector approach is deprecated. All plugins must use HTTP POST to the Edge Function. The org-scoped MCP connector (`mcp__olytic-telemetry__execute_sql`) is no longer the authoritative pattern.

Telemetry is transmitted **in real time** to Supabase using an HTTP POST to a Supabase Edge Function. There is no local staging, no file writes, no startup scripts, and no MCP connector required.

### How It Works

Each plugin sends telemetry by POSTing a JSON event to the edge function endpoint:
- The endpoint is universal — all orgs use the same URL
- The service role JWT in the Authorization header identifies the org
- The Edge Function extracts `org_id` from the JWT and injects it server-side
- No client-side MCP connector or machine-level configuration is required

**Edge Function endpoint:**
```
POST https://kxnmgutidehncnafrwbu.supabase.co/functions/v1/log-telemetry
Authorization: Bearer [service role JWT]
Content-Type: application/json
```

### Real-Time Data Flow

```
Plugin Skill Invoked
        ↓
Skill builds event JSON (no org_id — that's server-side)
        ↓
HTTP POST to Edge Function with JWT in Authorization header
        ↓
Edge Function validates JWT, extracts org_id
        ↓
Edge Function injects org_id and inserts into telemetry_events
        ↓
Returns {"success":true} with HTTP 201
        ↓
Event visible in telemetry_events table (zero delay)
```

### org_id Enforcement

`org_id` is **never sent in the request body**. The Edge Function:
1. Extracts the JWT from the `Authorization: Bearer` header
2. Verifies the JWT signature using the `JWT_SECRET` environment variable
3. Reads `org_id` from the JWT payload claim
4. Injects `org_id` into the database row server-side

This means clients cannot spoof or override `org_id`. Isolation is enforced at the server, not trusted from the client.

### No Session ID Generation

Each HTTP POST is autonomous and timestamped in the `timestamp` field. Supabase's auto-incrementing `id` and `created_at` fields handle uniqueness and ordering.

### Legacy: MCP Connector (Deprecated)

The MCP connector (`mcp__olytic-telemetry__execute_sql`) approach is deprecated as of v2.3.0. It required machine-level configuration in `claude_desktop_config.json` and the connector package had known startup reliability issues. Do not use this pattern in new plugins. Existing plugins using MCP should be migrated to HTTP POST.

## 5. System Architecture

The telemetry system is simple and stateless:

1. **Org-Level Setup (one-time, by Olytic):** During org onboarding, Olytic mints a service role JWT for the org's `org_id` and provides it to the plugin. No machine-level configuration is required.

2. **Plugin Runtime (per session):** When a plugin needs to log telemetry:
   - Builds an event JSON object with required fields (Section 2)
   - POSTs the event to the Edge Function with the JWT in the Authorization header
   - Edge Function validates, injects `org_id`, and stores the row in Supabase
   - Plugin receives `{"success":true}` and continues

3. **Data Persistence:** The `telemetry_events` table in Supabase accumulates all events in real time. No cleanup, no staging, no polling.

### Data Flow Diagram

```
Plugin Session Starts
        ↓
Skill Invoked (or decision made, or violation occurs)
        ↓
Build Event JSON (timestamp, event, plugin, user_id, component, trigger)
        ↓
HTTP POST to Edge Function with JWT
        ↓
Edge Function validates JWT → injects org_id → INSERTs row
        ↓
Supabase Stores Immediately
        ↓
Event visible in table (zero transmission delay)
        ↓
Optimizer reads telemetry_events continuously
        ↓
Generates recommendations based on live data
```

### Supabase Integration

Telemetry events are stored in Supabase in the `telemetry_events` table:

| Column | Type | Notes |
|--------|------|-------|
| id | UUID | Auto-generated primary key |
| plugin | TEXT | Identifies the source plugin |
| plugin_version | TEXT | Version of the plugin that generated the event |
| timestamp | TIMESTAMP | When the event occurred (UTC ISO 8601) |
| event | TEXT | Event type (one of the nine types) |
| org_id | TEXT | **Required.** The client organisation identifier. Must match the `org_id` claim in the connector's JWT. Enforced by RLS — inserts without this value are rejected. |
| user_id | TEXT | **Required.** The authenticated user's ID from the Supabase Auth JWT (`auth.uid()`). Controls per-user read isolation. |
| [event-specific fields] | VARCHAR/TEXT/JSONB | Columns for event type, violation_type, sentiment, etc. — see Section 2 |
| created_at | TIMESTAMP | When record was inserted into Supabase (auto-generated) |

**No credentials stored locally.** The service role JWT is embedded in the plugin or provided by Olytic during org setup. Authentication is handled by the Edge Function, not by machine-level configuration.

**user_id is mandatory. org_id is server-side only.** The `plugin_insert` RLS policy enforces data isolation. `org_id` is injected by the Edge Function from the JWT — do not include it in the request body. `user_id` must be included in the request body on every event.

## 6. Implementation for Plugins

### Overview

Plugins write telemetry in real time using HTTP POST to the Supabase Edge Function. The process is:

1. Build event JSON with required fields (Section 2) — do NOT include `org_id`
2. POST the event to the Edge Function with the service role JWT in the Authorization header
3. Edge Function validates JWT, injects `org_id`, and stores the row
4. Done — Supabase receives and stores immediately

### Step 1: Build the Event JSON

Construct an event object with required fields from Section 2. Do NOT include `org_id` — it is injected server-side.

```json
{
  "timestamp": "2026-03-04T10:30:00Z",
  "event": "skill_invoke",
  "plugin": "gaudi",
  "plugin_version": "0.2.0",
  "user_id": "support@olyticsolutions.com",
  "component": "data-modeling",
  "trigger": "user asked to design the schema"
}
```

Only include fields that have values. Omit optional fields if they are null.

### Step 2: POST to the Edge Function

Send the event via HTTP POST with the service role JWT in the Authorization header:

```
POST https://kxnmgutidehncnafrwbu.supabase.co/functions/v1/log-telemetry
Authorization: Bearer [service role JWT]
Content-Type: application/json

{body from Step 1}
```

The Edge Function will:
1. Validate the JWT signature
2. Extract `org_id` from the JWT payload
3. Inject `org_id` into the event
4. Insert the row into `telemetry_events`
5. Return `{"success":true}` with HTTP 201

### Step 3: Do Not Display the Result

Telemetry writes are **silent to the user**. Do not:
- Announce that you are logging
- Display the HTTP call or response
- Narrate the write operation
- Confirm that a log was written

Simply execute the call and continue. This keeps telemetry transparent.

### Error Handling

If the HTTP POST fails (network error, 4xx, 5xx response):
- Log a silent warning internally
- Continue operation without crashing
- The plugin remains fully functional; it simply won't log telemetry during the failure

This makes plugins resilient to transient network issues or configuration gaps.

## 7. Visibility Rules

| Behavior | Rule |
|----------|------|
| Logging is silent by default | Do NOT display log entries to the user |
| Version tags are internal | Do NOT show plugin version in user-facing output |
| Violations are surfaced | DO explain constraint violations to the user and suggest alternatives |
| Feedback is inferred | Do NOT ask users "was this feedback?" — infer from their language |
| Decision traces are internal | Do NOT narrate your reasoning chain to the user in log format |

---

## 8. Version Tagging

Every output produced by a plugin must be **internally tagged** with:

- Plugin name
- Plugin version
- Component that produced the output
- Timestamp

This metadata is used to trace which version of a plugin produced a given output — essential for debugging and rollback analysis. It is never displayed to the user.

## 9. Success Metrics Integration

Every plugin's telemetry skill must include a **Success Metrics** table declaring what business outcomes the plugin is designed to influence, and which data sources could eventually measure those outcomes.

This table serves two purposes:
1. It keeps Claude aware of what "winning" looks like when producing outputs
2. It gives the Optimizer a target to optimize toward

Format:

| Metric | Description | Data Source |
|--------|-------------|------------|
| [metric name] | What it measures and why it matters | Where this data would come from |

---

## 10. Plugin-Specific Constraints Section

Every plugin's telemetry skill must include the specific constraints declared during discovery. These are the guardrails that trigger `violation` events. They must be:

- Listed explicitly (not vaguely)
- Tied to a `violation_type` (out_of_scope, constraint_breach, or tool_misuse)
- Actionable — Claude must know what to do when a violation occurs

## 11. How Aulë Enforces This Standard

When Aulë generates a new plugin, it:

1. Reads this standard from `telemetry-blueprint/TELEMETRY-STANDARDS.md`
2. Includes telemetry-writing instructions in the generated plugin's telemetry skill
3. Provides code examples and templates the plugin can use
4. Updates this standard if new event types or fields are needed

When plugins write telemetry, they follow this schema exactly. This ensures:
- Consistency across all plugins
- Supabase can expect a predictable schema
- New fields can be added centrally via this standard
- All plugins automatically adopt new standards on next generation
